
### 📊 方案优劣讨论

根据您的需求，尤其是“实时掌握广告投放表现”和“及时发现异常”这两个核心点，我会首先认可您的第二个方案，即**实时检测方案**。

| 特性        | 方案一：定时运行规则检测 | 方案二：实时检测（每次拉取后） |
| --------- | ------------ | --------------- |
| **实时性**   | 较差，存在“时间真空期” | 优秀，数据拉取后立即处理    |
| **异常发现**  | 滞后，可能导致损失扩大  | 及时，能迅速预警并止损     |
| **数据新鲜度** | 依赖定时任务频率     | 极高，基于最新拉取数据     |
| **资源消耗**  | 集中在定时任务执行时   | 分散在每次数据拉取后      |
| **复杂性**   | 相对较低，基于已存数据  | 相对较高，需要处理实时流    |

#### 1. ⚙️ 设计监控配置表

首先，我们需要一个表来存储所有的监控规则。这个表将定义哪些指标在什么条件下触发告警。

**表结构示例 (`ad_monitor_rules`)**:

```sql
CREATE TABLE `ad_monitor_rules` (
  `id` INT UNSIGNED NOT NULL AUTO_INCREMENT,
  `rule_name` VARCHAR(100) NOT NULL COMMENT '规则名称',
  `project_id` INT UNSIGNED DEFAULT NULL COMMENT '关联游戏ID (可为NULL，表示全局规则)',
  `appid` INT UNSIGNED DEFAULT NULL COMMENT '关联APPID (可为NULL)',
  `os` TINYINT UNSIGNED DEFAULT NULL COMMENT '操作系统 (可为NULL)',
  `country_code` VARCHAR(64) DEFAULT NULL COMMENT '国家编码 (可为NULL)',
  `partner_id` INT UNSIGNED DEFAULT NULL COMMENT '广告渠道ID (可为NULL)',
  `metric` VARCHAR(50) NOT NULL COMMENT '监控指标 (如: cost, click, install, cpa, roi)',
  `monitor_type` ENUM('threshold', 'fluctuation', 'comparison') NOT NULL COMMENT '监控类型: 阈值, 波动, 对比',
  `threshold_value` DECIMAL(10,2) DEFAULT NULL COMMENT '阈值 (如: cost > 1000)',
  `threshold_operator` ENUM('>', '<', '>=', '<=', '=') DEFAULT NULL COMMENT '阈值操作符',
  `fluctuation_percent` DECIMAL(5,2) DEFAULT NULL COMMENT '波动百分比 (如: 20表示20%波动)',
  `fluctuation_period` INT DEFAULT NULL COMMENT '波动周期 (分钟/小时/天, 结合cron)',
  `comparison_metric` VARCHAR(50) DEFAULT NULL COMMENT '对比指标 (如: ROI)',
  `comparison_operator` ENUM('>', '<', '>=', '<=', '=') DEFAULT NULL COMMENT '对比操作符',
  `comparison_value` DECIMAL(10,2) DEFAULT NULL COMMENT '对比值',
  `comparison_dimension` VARCHAR(100) DEFAULT NULL COMMENT '对比维度 (如: campaignid, adgroupid)',
  `status` TINYINT(1) NOT NULL DEFAULT '1' COMMENT '规则状态: 1-启用, 0-禁用',
  `alert_level` ENUM('info', 'warning', 'critical') NOT NULL DEFAULT 'warning' COMMENT '告警级别',
  `alert_channels` VARCHAR(255) NOT NULL DEFAULT 'email' COMMENT '告警通知渠道 (逗号分隔)',
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  INDEX `idx_project_id` (`project_id`),
  INDEX `idx_metric` (`metric`)

CREATE TABLE `ad_monitor_alerts_log` (
  `id` INT UNSIGNED NOT NULL AUTO_INCREMENT,
  `rule_id` INT UNSIGNED NOT NULL COMMENT '关联的规则ID',
  `alert_level` ENUM('info', 'warning', 'critical') NOT NULL,
  `alert_message` TEXT NOT NULL COMMENT '告警消息',
  `alert_data` JSON DEFAULT NULL COMMENT '触发告警的原始数据或聚合数据',
  `is_resolved` TINYINT(1) NOT NULL DEFAULT '0' COMMENT '是否已解决: 1-是, 0-否',
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

  INDEX `idx_rule_id` (`rule_id`),
  INDEX `idx_created_at` (`created_at`)

```

**配置说明**:

- **`metric`**: 明确要监控的指标，例如 `cost`, `click`, `install`, `convert`, `exposure`。同时，您可能需要监控一些衍生指标，如 CPA (cost/install), ROI (charge/cost)。
    
- **`monitor_type`**:
    
    - **`threshold`**: 简单阈值告警，如 `cost > 1000`。
        
    - **`fluctuation`**: 波动告警，与前一个周期（如：上一个小时，昨天同时间）相比，指标变化超过一定百分比。
        
    - **`comparison`**: 对比告警，例如不同 `campaignid` 之间的 CPA 差异过大。
        
- **维度字段**: `project_id`, `appid`, `os`, `country_code`, `partner_id`, `campaignid`, `adgroupid` 等，用于精细化告警。可为 `NULL` 表示对所有该维度的数据生效（全局规则）。
    
- **`alert_channels`**: 支持多种告警方式，如邮件、短信、钉钉/企业微信消息等。

#### 2. 🧹 数据清洗与聚合模块 (PHP Worker)

此模块负责从 `ad_cost` 表中提取最新数据，进行必要的清洗、转换和聚合，然后将处理后的数据发送到消息队列。

**实现步骤**:

1. **数据拉取触发**:
    
    - **方案 A (推荐 - 实时性高):** 修改现有定时脚本。在数据成功写入 `ad_cost` 表后，立即触发一个事件（例如，向 MQ 发送一条消息），包含新写入数据的 `id` 范围或关键标识。
        
    - **方案 B (定时扫描):** 运行一个独立的 PHP Cron Job，每隔N分钟（例如5-15分钟）扫描 `ad_cost` 表，查找 `created_at` 或 `updated_at` 在上次扫描之后的新增/更新数据。
        
        ```php
        // 示例：定时扫描
        // 记录上次扫描时间
        $lastScanTime = file_get_contents('last_scan_time.txt');
        if (!$lastScanTime) $lastScanTime = date('Y-m-d H:i:s', strtotime('-1 hour')); // 首次运行或文件丢失，回溯1小时
        
        $pdo = new PDO(...); // 您的MySQL连接
        $stmt = $pdo->prepare("SELECT * FROM ad_cost WHERE updated_at > :lastScanTime ORDER BY updated_at ASC");
        $stmt->execute([':lastScanTime' => $lastScanTime]);
        $newRecords = $stmt->fetchAll(PDO::FETCH_ASSOC);
        
        // 更新扫描时间
        if (!empty($newRecords)) {
            $lastRecord = end($newRecords);
            file_put_contents('last_scan_time.txt', $lastRecord['updated_at']);
        }
        ```
        
2. **数据清洗**: 确保数据的有效性、一致性。例如，处理 `NULL` 值、格式转换等。
    
3. **数据聚合**: 根据监控需求，将原始数据聚合到不同的维度和时间粒度。
    
    - **维度**: `project_id`, `appid`, `os`, `country_code`, `partner_id`, `campaignid`, `adgroupid`。
        
    - **时间粒度**: 根据实时性要求，可以聚合到小时、半小时，甚至15分钟。
        
    - **计算衍生指标**: 计算 CPA (`cost/install`)、ROI (`charge/cost`)、CTR (`click/exposure`) 等。
        
    
    ```php
    // 示例：PHP中聚合数据
    $aggregatedData = [];
    foreach ($newRecords as $record) {
        $key = implode('_', [
            $record['date'],
            $record['project_id'],
            $record['appid'],
            $record['os'],
            $record['country_code'],
            $record['partner_id'],
            $record['campaignid'],
            // ... 可以根据监控粒度添加更多维度，如 adgroupid, creativeid
            // 实时监控通常不精确到 creativeid，因为数据量太大
        ]);
    
        if (!isset($aggregatedData[$key])) {
            $aggregatedData[$key] = [
                'date' => $record['date'],
                'project_id' => $record['project_id'],
                'appid' => $record['appid'],
                'os' => $record['os'],
                'country_code' => $record['country_code'],
                'partner_id' => $record['partner_id'],
                'campaignid' => $record['campaignid'],
                'cost' => 0,
                'charge' => 0,
                'install' => 0,
                'convert' => 0,
                'click' => 0,
                'exposure' => 0,
                // ... 其他指标
            ];
        }
        $aggregatedData[$key]['cost'] += $record['cost'];
        $aggregatedData[$key]['charge'] += $record['charge'];
        $aggregatedData[$key]['install'] += $record['install'];
        $aggregatedData[$key]['convert'] += $record['convert'];
        $aggregatedData[$key]['click'] += $record['click'];
        $aggregatedData[$key]['exposure'] += $record['exposure'];
        // ...
    }
    
    // 计算衍生指标
    foreach ($aggregatedData as &$data) {
        $data['cpa'] = ($data['install'] > 0) ? ($data['cost'] / $data['install']) : 0;
        $data['roi'] = ($data['cost'] > 0) ? ($data['charge'] / $data['cost']) : 0;
        $data['ctr'] = ($data['exposure'] > 0) ? ($data['click'] / $data['exposure']) : 0;
        // ...
    }
    ```
    
4. **发送到消息队列 (MQ)**: 将聚合后的数据作为消息发送到 MQ (如 RabbitMQ, Kafka)。为不同的时间粒度或维度的数据使用不同的 Topic/Queue，以便下游消费者能够灵活处理。
    
    ```php
    // 示例：发送到MQ
    $mqProducer = new MQProducer(); // 您的MQ生产者实例
    foreach ($aggregatedData as $data) {
        $mqProducer->publish('ad_metrics_aggregated', json_encode($data));
    }
    ```
    
    - **MQ 的作用**: 解耦、削峰、异步处理，确保数据处理的稳定性和可靠性。

#### 3. 🚨 监控规则检测模块 (PHP Worker)

这是一个常驻进程的 PHP Worker，从 MQ 消费聚合数据，并根据预设的规则进行检测。

**实现步骤**:

1. **消费 MQ 消息**: 监听 `ad_metrics_aggregated` 队列，获取聚合后的数据。
    
    ```php
    $mqConsumer = new MQConsumer(); // 您的MQ消费者实例
    $mqConsumer->consume('ad_metrics_aggregated', function($message) {
        $data = json_decode($message->body, true);
        // ... 进行规则检测
    });
    ```
    
2. **加载监控规则**:
    
    - **Redis 缓存**: 监控规则应从 `ad_monitor_rules` 表加载到 Redis 中进行缓存，以减少数据库查询压力并提高检测速度。当规则配置发生变化时，需要有机制（如管理后台操作后触发消息）更新 Redis 缓存。
        
    - **规则结构**: 缓存的规则应便于按 `project_id`, `appid`, `metric` 等维度快速检索。
        
    
    ```php
    // 示例：从Redis获取规则
    $redis = new Redis(); // 您的Redis连接
    $rules = $redis->get('ad_monitor_rules_cache');
    if (!$rules) {
        // 从MySQL加载并缓存
        $pdo = new PDO(...);
        $stmt = $pdo->query("SELECT * FROM ad_monitor_rules WHERE status = 1");
        $rules = $stmt->fetchAll(PDO::FETCH_ASSOC);
        $redis->set('ad_monitor_rules_cache', json_encode($rules), 3600); // 缓存1小时
    } else {
        $rules = json_decode($rules, true);
    }
    ```
    
3. **规则匹配与检测**:
    
    - 对每条接收到的聚合数据，遍历所有相关的监控规则。
        
    - **维度匹配**: 优先匹配具体维度（如 `project_id=X, appid=Y`）的规则，如果没有，则匹配更通用的规则（如 `project_id=X`，或全局规则）。
        
    - **指标计算**: 根据 `metric` 字段从聚合数据中获取对应的值。
        
    - **执行检测逻辑**:
        
        - **阈值检测**: `if ($data[$rule['metric']] {$rule['threshold_operator']} $rule['threshold_value']) { ... }`
            
        - **波动检测**: 需要存储历史数据。可以利用 Redis 存储每个维度-指标在过去N个周期的数据（例如，`ad:history:campaignid:123:cost:2026032015` -> 15点的数据）。
            
            - 计算当前值与前一个周期值的百分比变化：`((current_value - previous_value) / previous_value) * 100%`。
                
            - 判断是否超过 `fluctuation_percent`。
                
        - **对比检测**: 比较不同维度（如 `campaignid`）下的同一指标。可能需要在一个批次中处理多个 `campaignid` 的数据，或者从 Redis 获取其他 `campaignid` 的历史/最新数据进行比较。
            
    
    ```php
    // 示例：PHP规则检测逻辑骨架
    function checkRules($data, $rules) {
        foreach ($rules as $rule) {
            // 1. 维度匹配 (简化示例，实际应更复杂，考虑NULL和优先级)
            if (($rule['project_id'] !== null && $rule['project_id'] != $data['project_id']) ||
                ($rule['appid'] !== null && $rule['appid'] != $data['appid']) ||
                // ... 其他维度匹配
                ($rule['metric'] != 'all' && $rule['metric'] != $data['metric_being_checked_from_aggregated_data'])) { // 假设 data 中有一个字段表示正在检查的指标
                continue; // 不匹配的规则跳过
            }
    
            $metricValue = $data[$rule['metric']] ?? null;
            if ($metricValue === null) continue; // 指标不存在
    
            $isAlert = false;
            $alertMessage = "";
    
            switch ($rule['monitor_type']) {
                case 'threshold':
                    if (eval("return {$metricValue} {$rule['threshold_operator']} {$rule['threshold_value']};")) {
                        $isAlert = true;
                        $alertMessage = "指标 {$rule['metric']} ({$metricValue}) 触发表 {$rule['threshold_operator']} {$rule['threshold_value']} 阈值。";
                    }
                    break;
                case 'fluctuation':
                    // 需要从 Redis 获取上一个周期的数据进行比较
                    $previousPeriodValue = getPreviousPeriodMetricValue($data, $rule['metric'], $rule['fluctuation_period']); // 假定一个获取历史数据的函数
                    if ($previousPeriodValue !== null && $previousPeriodValue != 0) {
                        $fluctuation = (($metricValue - $previousPeriodValue) / $previousPeriodValue) * 100;
                        if (abs($fluctuation) > $rule['fluctuation_percent']) {
                            $isAlert = true;
                            $alertMessage = "指标 {$rule['metric']} ({$metricValue}) 较上一周期 ({$previousPeriodValue}) 波动 {$fluctuation}%，超过 {$rule['fluctuation_percent']}%。";
                        }
                    }
                    // 存储当前数据到 Redis，供下个周期使用
                    saveCurrentMetricValue($data, $rule['metric']);
                    break;
                case 'comparison':
                    // 假设比较 CPA：与同 campaignid 下其他 adgroupid 的平均 CPA 比较
                    // 或与全局平均 CPA 比较。这需要更复杂的逻辑，可能需要查询 Redis 或 MySQL
                    $comparisonValue = getComparisonMetricValue($data, $rule); // 假定一个获取对比数据的函数
                    if ($comparisonValue !== null && eval("return {$metricValue} {$rule['comparison_operator']} {$comparisonValue};")) {
                        $isAlert = true;
                        $alertMessage = "指标 {$rule['metric']} ({$metricValue}) 与 {$rule['comparison_dimension']} 的 {$rule['comparison_metric']} ({$comparisonValue}) 相比，触发表 {$rule['comparison_operator']} {$rule['comparison_value']} 规则。";
                    }
                    break;
            }
    
            if ($isAlert) {
                // 发现异常，发送告警消息到告警MQ
                sendAlertToMQ($rule, $data, $alertMessage);
            }
        }
    }
    ```
    
4. **发送告警消息**: 如果检测到异常，将告警信息（包括规则ID、告警级别、告警消息、原始数据等）发送到另一个专门的 MQ 队列 (如 `ad_monitor_alerts`)。

#### 4. 📢 异常告警机制 (PHP Worker)

此模块负责从告警 MQ 消费消息，并根据配置的告警渠道发送通知。

**实现步骤**:

1. **消费告警 MQ 消息**: 监听 `ad_monitor_alerts` 队列。
    
2. **解析告警信息**: 获取告警规则ID、告警级别、告警消息和相关数据。
    
3. **查询告警渠道**: 根据告警信息中的 `alert_channels` (可能来自 `ad_monitor_rules` 表) 确定需要发送通知的渠道。
    
4. **发送通知**:
    
    - **邮件**: 使用 PHPMailer 或 Symfony Mailer 等库发送邮件。
        
    - **短信**: 集成第三方短信服务 API。
        
    - **即时通讯**: 集成企业微信/钉钉/飞书等群机器人 API，发送富文本消息。
        
    - **工单系统**: 如果有工单系统，可以自动创建工单。
        
5. **记录告警日志**: 将告警详情记录到 `ad_monitor_alerts_log` 表中，便于后续审计和查看。

#### 5. 🛠️ 技术栈整合

- **PHP**: 作为主要的开发语言，编写数据清洗、聚合、规则检测和告警发送的 Worker 脚本。建议使用 [Supervisor](https://www.google.com/url?q=http%3A%2F%2Fsupervisord.org%2F "http://supervisord.org/") 来管理和守护这些 PHP Worker 进程，确保它们稳定运行。
    
- **MySQL**:
    
    - `ad_cost`: 原始数据源。
        
    - `ad_monitor_rules`: 存储监控规则配置。
        
    - `ad_monitor_alerts_log`: 存储告警历史记录。
        
- **Redis**:
    
    - **监控规则缓存**: 缓存 `ad_monitor_rules` 表的数据，提高检测效率。
        
    - **历史数据存储**: 用于波动检测，存储关键指标在过去N个时间窗口的数据。例如，存储 `campaignid:metric:hour` 的聚合值，key 为 `ad:hourly:cost:campaignid:123:2026032015`，value 为 `1000.00`。
        
    - **计数器/限流**: 如果需要对某个维度（如某个campaign）的告警进行限流，防止短时间内发送大量重复告警。
        
- **MQ (Message Queue)**:
    
    - **数据聚合队列**: (`ad_metrics_aggregated`) 从数据清洗聚合模块到规则检测模块。
        
    - **告警队列**: (`ad_monitor_alerts`) 从规则检测模块到告警服务模块。
        
    - **选型**: RabbitMQ (功能丰富，成熟稳定), Kafka (高吞吐量，分布式)，根据您的团队经验和系统规模选择。
