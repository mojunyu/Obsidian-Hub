# Caveman 使用文档

  

`caveman` 是一个超精简沟通模式。目标：减少冗词，保留技术准确性，降低 token 消耗。

  

## 1. 快速开始

  

- 开启（默认强度 `full`）：`/caveman`

- 指定强度：`/caveman lite`、`/caveman full`、`/caveman ultra`

- 停止模式：`stop caveman` 或 `normal mode`

  

说明：

- 模式开启后会持续生效，不会因为对话轮次增加自动退出。
	
- 仅在你明确说停止时才恢复普通表达。

  

## 2. 何时触发

  

手动触发关键词（示例）：

- `caveman mode`

- `talk like caveman`

- `use caveman`

- `less tokens`

- `be brief`

- `/caveman`

  

自动触发：

- 当你明确要求“更省 token / 更简短”时，会自动进入 caveman 风格。

  

## 3. 强度档位说明

  

### lite

- 去掉客套和赘词

- 保持完整句子和正常语法

- 适合：跨团队沟通、对外文档草稿

  

### full（默认）

- 去掉冠词和填充词

- 允许短句、片段句

- 适合：日常开发沟通、排障对话

  

### ultra

- 进一步缩写（如 `DB`、`auth`、`req/res`、`fn`）

- 多用因果箭头（`X -> Y`）

- 适合：高频迭代、极限压缩上下文

  

### wenyan-lite

- 半文言风格，仍保留基本可读结构

  

### wenyan-full

- 高压缩文言风格，语句更短更“古风”

  

### wenyan-ultra

- 文言极简风格，压缩比最高

  

## 4. 输出规则（核心）

  

`caveman` 输出遵循：

- 删冗词：去掉“just/really/basically/actually/simply”等填充

- 删客套：减少“当然可以/很高兴帮你”类开场

- 术语不变：技术名词、函数名、错误文本保持准确

- 代码不改：代码块原样表达，不做“口语化改写”

  

推荐句式模板：

- `[问题] [动作] [原因]。 [下一步]。`

  

示例：

- 普通：这个接口慢是因为每次都新建数据库连接，建议用连接池。

- caveman：接口慢。每次 req 新建 DB 连接。上连接池复用。

  

## 5. 自动“降回清晰模式”的场景

  

为避免误解，以下场景会自动降低 caveman 压缩力度，优先清晰表达：

  

- 安全警告（数据泄露、权限风险）

- 不可逆操作确认（删库、批量删除、覆写）

- 多步骤关键操作（顺序错会出事故）

- 你明确要求“解释清楚一点”

  

处理完关键说明后，再恢复 caveman 风格。

  

## 6. 实战示例（同一问题，不同强度）

  

问题：为什么 React 组件频繁重渲染？

  

- lite：组件重渲染因为每次渲染都创建了新的对象引用，建议用 `useMemo` 缓存。

- full：每次 render 新对象引用。prop 变新引用 -> 重渲染。用 `useMemo`。

- ultra：inline obj -> new ref -> re-render。`useMemo`。

  

## 7. 团队使用建议

  

- 默认用 `full`：压缩和可读性平衡最好。

- 评审/复盘文档输出前，切 `lite`：更正式，便于转存。

- 仅在内部高频排障时用 `ultra`：效率高，但新同学阅读成本更高。

- 涉及生产风险操作时，临时切普通模式：避免误操作。

  

## 8. 常见问答

  

Q: 开启一次后会自动关闭吗？

A: 不会。需手动 `stop caveman` 或 `normal mode`。

  

Q: 会影响代码内容吗？

A: 不会。代码、命令、错误文本应保持技术准确和原意。

  

Q: 想更短但不想太“碎”怎么办？

A: 用 `lite`，通常最稳。
